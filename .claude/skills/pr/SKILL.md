description: "現在のブランチをレビュー観点で評価し、適切なら説明文を生成、過大なら分割方針を先に提示する"
---

現在のブランチをレビュー観点で評価し、PRとして妥当な範囲かを先に判断する。妥当なら説明文を生成し、過大なら説明文は保留して分割方針を提示する。

1. 全体像を把握する
   - `git log main..HEAD --oneline` でコミット一覧を確認する
   - `git diff main...HEAD --stat` で変更規模（ファイル数・行数）を把握する
   - `git diff main...HEAD` で変更内容を確認する
2. レビュー負荷を評価する
   - 観点: レビュー者が正確に追えるサイズか / 関心が一つにまとまっているか / コミット履歴が単位として成立しているか
   - 最終的な判断基準は「レビュー者が正確にレビューできる範囲か」
   - 目安（厳密な閾値ではない）:
     - 変更ファイルが少なく（〜10ファイル程度）変更行数も限定的（〜300行程度）なら、関心が複数あっても1PRに収めて良い
     - 変更行数が大きい（500行〜）、または独立した関心が明らかに混在している場合は分割を検討する
     - 数千行を超えると正確なレビューはほぼ不可能だと認識する
3. 分割が望ましいと判断した場合
   - PR説明文は生成しない
   - 評価の根拠（規模・関心の混在状況）と分割方針案（どの範囲を1PRとするか、依存順序、ブランチ戦略）を提示する
   - ユーザーの判断（このまま進める / 分割する）を待つ
4. 分割不要、または「このまま進める」と判断された場合に説明文を生成する
   - 冒頭にレビュー評価の要点を1〜2行で添える
   - 以下の形式で出力する:

## Summary
- （変更の概要を箇条書き）

## Changes
- （具体的な変更点を箇条書き）

## Test plan
- （確認すべきことを箇条書き）
