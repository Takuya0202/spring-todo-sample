---
description: "未ステージも含めた変更を関心ごとに分割コミット案として提示し、承認後に順にコミットする"
---

変更内容を関心ごとに分け、conventional commits形式で順にコミットする。1つの関心しかない・差分が十分小さい場合は分割せずそのままコミットする。

1. 現状を把握する
   - `git status` でステージ済み・未ステージ・追跡外を確認する
   - `git diff` および `git diff --staged` で変更内容を把握する
2. 変更を関心ごとに分類する
   - 観点例: 機能・関心領域、レイヤー、変更種別（feat / fix / refactor / docs / chore 等）、実装とテスト・ドキュメントの関係
   - 「1コミットだけ revert しても他の単位が破綻しないか」を分割の基準にする
3. 分割案をユーザーに提示する
   - 単位ごとに「含めるファイル一覧 / 種別 / コミットメッセージ案」をまとめて提示する
   - 分割不要と判断したときはその旨と理由を伝える
4. ユーザーが承認したら順にコミットする
   - 単位ごとに `git add <files>` で個別にステージする（`git add .` / `git add -A` は使わない）
   - 「なぜその変更をしたか」が伝わる日本語のコミットメッセージを1行（最長72文字）で作成する
   - `git commit -m "..."` を実行する（Co-authored-byは自動付与）
   - 途中で意図しない差分が混じる等の異常があれば止めてユーザーに報告する
5. push はユーザーが明示的に指示したときだけ実行する
